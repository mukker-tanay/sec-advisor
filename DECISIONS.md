# Decisions

One line per non-trivial choice: what was picked, what was passed over, why.

- **NVD REST API over bulk JSON feed** — fetched 500 records via a single API call instead of downloading NVD's full yearly bulk dump. Bulk feeds are tens-to-hundreds of MB for data we'd just slice down anyway; the API lets us ask for exactly the page size we need in one request.
- **No literal `remediation` field** — NVD doesn't provide structured fix-it text. Rather than fabricate one, we keep the real signals (CVSS severity/score, CWE weakness IDs, reference URLs) and let the RAG generation step compose remediation guidance from those at answer time, citing sources. Storing a fake field would mean inventing data that isn't there.
- **Committed `data/` to git instead of gitignoring it** — the snapshot is small (500 records, ~1-2MB) and is the deliberate "static, not live" design choice; committing it makes the repo self-contained so a clone can run the demo without hitting the NVD API again.
- **uv for Python env/package management** — used throughout instead of bare pip/venv, per personal workflow preference.
- **Point IDs derived from a hash of `cve_id`** — Qdrant requires int or UUID point IDs, not arbitrary strings. Hashing `cve_id` into a stable int means re-running the embed script upserts (updates) existing records instead of duplicating them, with no separate ID-tracking needed.
- **One Qdrant collection, one chunk per CVE, cosine distance** — confirmed via raw similarity search that this granularity retrieves relevant results without needing finer chunking.
- **Groq (llama-3.1-8b-instant) over Gemini** — fast inference, better for live demos; free tier, no card.
- **Two-layer grounding: score threshold + LLM-checked relevance** — a similarity-score cutoff (0.2) alone isn't reliable, since semantically-similar-but-wrong queries (e.g. "OpenSSL 3.0" against 1999-era CVE data) can score almost as high as genuinely relevant ones. The threshold only filters obvious noise; the prompt additionally instructs the LLM to verify retrieved CVEs actually address the question and refuse ("I don't have an advisory on that") if not, even when something was retrieved.
- **Qdrant payload stored as `{page_content, metadata}`** — matches LangChain's `QdrantVectorStore` read format, so we use LangChain's retriever abstraction (the point of using LangChain) rather than bypassing it with raw qdrant-client calls.
- **Stack-exposure mode uses plain substring matching against `affected_products`, not vector similarity** — initially built with embedding search, but that produced false positives (e.g. "openssl 3.0" matching unrelated 1999 CVEs on semantic similarity alone). "Does my stack include X" is an exact membership check, not a natural-language question, so it needed a different retrieval strategy than the Q&A mode, not just a higher score threshold. No LLM call in this mode either — severity/CVSS/references are surfaced directly rather than generated.
- **Pydantic v2 schemas for parsing** — introduced strict validation schema in parsing (`parse_records.py`) to guard against API payload drift. If a record is invalid, we log a warning instead of crashing, keeping the ingestion resilient.
- **Paginated fetch with exponential backoff and jitter** — resolved NVD REST API rate limiting (403/429/503 errors) by paginating in 500-record chunks and executing a retry loop with progressive backoff and randomized jitter to guarantee script completion.
- **Incremental delta-syncing** — enabled delta syncing using `lastModStartDate`/`lastModEndDate` query parameters, scanning local records to find the latest modification timestamp to avoid redundant downloads.
- **LLM-as-a-judge evaluation suite** — created a standalone `evaluate.py` test harness to grade groundedness, refusal, and jailbreak safety using JSON formatting constraints, establishing a metric-driven performance baseline (69.2% overall accuracy).
- **Query rewriting & Qdrant pre-filtering (Hybrid Search)** — utilized ChatGroq to extract vendor/product names from raw query text and convert them into Qdrant payload filters, forcing similarity search to look only at the requested software. This eliminated semantic drift and raised overall RAG accuracy from 69.2% to 84.6%.
- **Local-only embeddings initialization** — configured `local_files_only=True` for HuggingFaceEmbeddings to prevent it from sending network check requests to the Hugging Face Hub, resolving connection hangs in restricted/offline execution environments.
- **Hugging Face Spaces Docker configuration** — wrote a custom `Dockerfile` utilizing `uv` for caching dependencies and exposing port `7860`, enabling one-click serverless deployment of the application and the local vector database.
